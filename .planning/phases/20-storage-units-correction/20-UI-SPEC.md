## STOR-03: FC/NFS Pool Input (StorageConfigForm.vue)

Add a new `NumberSliderInput` inside the FC/NFS branch of `StorageConfigForm.vue`. No dedicated `<template v-if="storageType === 'fc' || storageType === 'nfs'">` block exists today — one must be created, placed after the storage-type button row and before the capacity summary block.

```
Condition:  v-if="storageType === 'fc' || storageType === 'nfs'"
Component:  NumberSliderInput
v-model:    externalStorageUsableTiB   (domainField('externalStorageUsableTiB'))
:label:     t('host.externalStorageTiB')
:min:       1
:max:       2000
:step:      10
unit:       "TiB"
```

Default value for `externalStorageUsableTiB`: **100** TiB (from RESEARCH.md Assumption A3 and Zod schema default).

The capacity summary block (`<!-- Storage capacity summary -->`) must continue to render for FC/NFS; it will display the pool value as raw capacity and the pass-through net usable, both labeled TiB.

## Non-Goals for Phase 20

The following are explicitly out of scope. The executor must not implement any of these even if they seem related:

- No new components (NumberSliderInput already exists; no new wrappers needed)
- No layout changes (grid structure in StorageConfigForm unchanged)
- No color changes (green for net usable, blue for active type button — unchanged)
- No spacing changes (gap-4, space-y-4 patterns unchanged)
- No conversion arithmetic — adding a `× 1.0995` factor anywhere is a bug, not a feature
- No lossless URL migration — old `hostStorageTB` URLs degrade gracefully to the default via Zod `.strip()`; no migration code needed
- No changes to the NumberSliderInput component itself (only its props at call sites change)
- No changes to vSAN ESA FTT/RAID controls
- No changes to deduplication controls
- No dark mode additions beyond what already exists
